# 프로젝트 컨텍스트

## 1. 프로젝트 개요

이 저장소는 **시넥스 질문력 Co-Learner 프로젝트**를 위한 저장소다.

장기적으로는 학생의 질문력을 높여 주는 AI Co-Learner 서비스를 만드는 것이 목표다.  
다만 **현재 단계의 목표는 완성된 서비스 구현이 아니라**,  
그 서비스를 만들 수 있는 **AI 협업 조직(Agent 팀)을 설계하고 최소 실행 흐름을 검증하는 것**이다. :contentReference[oaicite:0]{index=0} :contentReference[oaicite:1]{index=1}

즉, 현재 프로젝트는 다음과 같이 정의한다.

> 우리는 최종 서비스를 바로 완성하는 것이 아니라,  
> 그 서비스를 만들어낼 수 있는 **AI Agent 팀**을 설계하고 구현한다.

---

## 2. 현재 단계

### 현재 단계
**1단계: AI 팀빌딩**

### 현재 단계의 목표
질문력 Co-Learner 서비스를 만들기 위한 AI 협업 조직을 구성하고,  
각 Agent가 역할에 따라 산출물을 만들어내는 최소 실행 흐름을 구현한다.

### 이 단계에서 성공으로 보는 기준
이 단계에서 성공은 완성된 서비스가 아니라 아래를 만족하는 것이다.

- AI 팀 구조가 명확히 정의되어 있다
- 각 Agent의 역할이 구체적이다
- 각 Agent의 입력값과 출력값이 정의되어 있다
- 협업 흐름이 논리적으로 연결되어 있다
- 팀이 최소 1회 end-to-end로 실행된다
- 최종적으로 서비스 제작 관련 산출물이 생성된다

---

## 3. 서비스 도메인

이 프로젝트가 다루는 서비스 도메인은 다음과 같다.

**질문력 향상을 위한 Question-Power Co-Learner**

### 대상 사용자
- 주 대상은 중학생
- 질문이 모호하거나 정보가 부족한 학생
- 무엇을 물어봐야 하는지 잘 모르는 학생

### 핵심 서비스 개념
이 서비스는 일반적인 지식 전달형 챗봇이 아니다.  
학생이 **자신의 질문을 더 정확하고 구체적으로 만들 수 있도록 돕는 질문 튜터**에 가깝다. :contentReference[oaicite:2]{index=2}

### 사용자가 느껴야 하는 핵심 경험
> “내 질문이 더 좋아졌네.”

### 질문 개선의 핵심 기준
질문은 아래 3가지 기준으로 더 좋아질 수 있다.

- **구체성**: 무엇에 대한 질문인지 드러나는가
- **맥락성**: 왜 지금 필요한 질문인지 드러나는가
- **목적성**: 어떤 도움을 원하는지 드러나는가

이 기준은 현재 MVP 설계 문서의 질문 진단 기준표를 따른다. :contentReference[oaicite:3]{index=3}

## 4. 왜 지금은 팀빌딩에 집중하는가

현재는 서비스 전체를 짧은 기간 안에 완성하는 것보다,  
그 서비스를 생산할 수 있는 구조를 먼저 만드는 것이 더 중요하다고 판단했다.

### 이유
- 일정이 짧다
- 서비스 전체 구현은 범위가 크다
- 지금 단계에서는 서비스의 완성도보다 **AI 협업 구조가 실제로 작동하는지**를 검증하는 것이 더 현실적이다

### 따라서 현재 단계에서는
- 서비스를 직접 완성하는 것보다
- 서비스를 만들기 위한 **생산 시스템**
- 역할 기반 **협업 구조**
- 최소 실행 가능한 **작업 흐름**
에 집중한다

---

## 5. 현재 팀 구조

현재 기준 AI 팀은 아래 5개 Agent로 구성한다.

1. **Product Planner Agent**
2. **Question Power Designer Agent**
3. **Quest Designer Agent**
4. **Growth Mapping Agent**
5. **Builder & QA Agent**

이 5개 Agent 구조를 현재 저장소의 기본 팀 구조로 삼는다.

---

## 6. Agent별 역할 정의

### 6-1. Product Planner Agent
**역할**  
프로젝트의 전체 방향과 MVP 범위를 정리하는 역할

**주요 업무**
- 문제 정의
- 프로젝트 목표 정리
- 핵심 사용자 정의
- MVP 범위 설정
- 제외할 범위 정리
- 기본 사용자 흐름 정리

**예상 산출물**
- 문제 정의
- 프로젝트 목표
- 타깃 사용자 정의
- MVP 범위
- 제외 범위
- 사용자 흐름 초안

---

### 6-2. Question Power Designer Agent
**역할**  
질문력 Agent 자체를 설계하는 역할

**주요 업무**
- 질문력 Agent 역할 정의
- 질문 개선 원칙 설계
- 금지사항 정의
- 시스템 프롬프트 초안 작성
- Few-shot 예시 작성

**예상 산출물**
- Agent 역할 정의서
- 핵심 원칙
- 금지사항
- 프롬프트 초안
- Few-shot 예시

---

### 6-3. Quest Designer Agent
**역할**  
챗봇 안에서 사용자가 수행할 퀘스트/인터랙션을 설계하는 역할

**설명**  
최종 서비스 범위가 줄어들더라도, 퀘스트는 여전히  
“사용자가 질문을 더 좋게 만들기 위해 수행하는 구조화된 상호작용”으로 의미가 있다.

**주요 업무**
- 퀘스트 유형 정의
- 퀘스트 흐름 정의
- 인터랙션 방식 설계
- 샘플 퀘스트 예시 작성

**예상 산출물**
- 퀘스트 유형
- 퀘스트 흐름
- 샘플 퀘스트
- 인터랙션 패턴

**예시 퀘스트**
- 더 좋은 질문 고르기
- 질문에서 빠진 요소 찾기
- 모호한 질문 다시 쓰기
- 특정 상황에 맞는 질문 만들기

---

### 6-4. Growth Mapping Agent
**역할**  
퀘스트 결과를 질문력 성장/점수/피드백으로 연결하는 역할

**주요 업무**
- 점수 규칙 설계
- 성장 단계 정의
- 피드백 문구 규칙 설계
- 결과 메시지 구조 정의

**예상 산출물**
- 점수 규칙표
- 성장 단계
- 피드백 템플릿
- 결과 메시지 규칙

---

### 6-5. Builder & QA Agent
**역할**  
앞선 Agent들의 결과를 통합하고 구현 및 검수 관련 산출물을 만드는 역할

**주요 업무**
- 상위 산출물 통합
- 구현 계획 정리
- 통합 메모 작성
- QA 체크리스트 작성
- 이슈 및 리스크 정리

**예상 산출물**
- 구현 계획
- 통합 메모
- QA 체크리스트
- QA 이슈
- 최종 요약 포인트

---

## 7. 협업 흐름

기본 협업 흐름은 아래와 같다.

1. **Product Planner Agent**
2. **Question Power Designer Agent**
3. **Quest Designer Agent**
4. **Growth Mapping Agent**
5. **Builder & QA Agent**

### 흐름의 의미
- Planner가 무엇을 만들지 정한다
- Question Designer가 질문력 Agent의 성격과 규칙을 만든다
- Quest Designer가 사용자가 어떤 상호작용을 하게 될지 설계한다
- Growth Mapping이 결과를 성장/피드백과 연결한다
- Builder & QA가 전체를 통합하고 검수한다

### 요약 흐름
**기획 → 질문력 Agent 설계 → 퀘스트 설계 → 성장 로직 설계 → 통합 및 QA**

## 8. 입력값/출력값 설계 원칙

이 프로젝트는 구조화된 파이프라인 방식으로 구현하는 것을 기본 원칙으로 한다.

### 기본 원칙
각 Agent는
- 구조화된 입력을 받고
- 자신의 역할에 맞게 처리한 뒤
- 구조화된 출력을 생성하고
- 다음 Agent에게 전달한다

### 권장 방식
- JSON 직렬화가 가능한 구조를 사용한다
- 사람이 읽어도 이해 가능한 결과물을 만든다
- 다음 Agent가 그대로 사용할 수 있도록 출력 형식을 일관되게 유지한다

현재 단계에서 저장소가 최종적으로 지원해야 하는 산출물은 아래와 같다.

- 질문 진단 기준
- 프롬프트 초안
- 퀘스트 구조

- QA 체크리스트

## 10. 현재 단계에서 하지 않아도 되는 것

현재 단계에서는 아래가 필수는 아니다.

- 완성형 프로덕션 챗봇
- 로그인/회원 기능
- DB 연동
- 고도화된 개인화 기능
- 장기 메모리 구조
- 완성도 높은 UI
- 복잡한 점수/레벨 시스템
- 완전 자율 멀티에이전트 운영 구조

시간이 허락하면 붙일 수 있지만, 지금 단계의 핵심 목표는 아니다.

---

## 11. 권장 기술 방향

### 최소 구현 방향
우선은 **순차 실행형 멀티에이전트 파이프라인**으로 구현한다.

즉,
- 각 Agent는 하나의 역할 단위
- 각 Agent는 입력을 받아 출력 생성
- 출력은 파일 또는 구조화된 결과물로 저장
- 마지막에 전체 결과를 요약한다

### 이후 확장 가능성
순차 파이프라인이 안정적으로 작동한 뒤에는 필요 시 아래를 확장할 수 있다.
- 경량 Orchestrator
- Streamlit 데모 UI
- QA 실패 시 1회 재실행 로직

---

## 12. 저장소의 역할

이 저장소는 크게 두 층을 지원해야 한다.

### Layer 1 — 팀 생산 레이어
AI 협업 팀이 산출물을 생산하는 레이어

### Layer 2 — 선택적 데모 레이어
필요 시 Streamlit 등으로 MVP 상호작용을 시연하는 레이어

현재 우선순위는 **Layer 1**이다.

---

## 13. 한 줄 요약

이 프로젝트는 질문력 Co-Learner 서비스를 위한 **1단계 AI 팀빌딩 프로젝트**이며,  
다섯 개의 전문 Agent가 협업하여 서비스 기획, 프롬프트, 퀘스트, 성장 로직, 구현/QA 산출물을 생산하는 구조를 만드는 것이 현재 목표다.
